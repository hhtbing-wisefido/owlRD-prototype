**创建日期**: 2025-11-20  
**目的**: 避免重复文档，保持项目记录清晰有序

## ❌ 常见问题示例

### 问题案例：重复的"自动更新"文档

**问题**:
```
❌ 自动更新规则.md      (7311字节) - 关于聊天记录的更新规则
❌ 自动更新说明.md      (4739字节) - 关于项目状态.json的脚本说明
```

**为什么这是问题**:
1. 名称太相似，容易混淆
2. 都叫"自动更新"但内容完全不同
3. 不清楚哪个文档是做什么的
4. 新成员无法快速理解

**解决方案**:
```
✅ 聊天记录管理规范.md   - 明确主题：聊天记录
✅ 状态自动更新说明.md   - 明确主题：项目状态脚本
```

## 📝 文档命名规范

### 1. 规范性文档

**格式**: `[主题]_[类型].md`

**类型选择**:
- **规范** - 描述规则和标准
- **说明** - 使用指南和操作步骤  
- **指南** - 详细的配置或部署指导

**示例**:
```
✅ 聊天记录管理规范.md
✅ 状态自动更新说明.md
✅ 局域网访问指南.md
✅ Git工作流规范.md
```

**反例**:
```
❌ 管理规范.md          - 太宽泛，管理什么？
❌ 自动更新.md          - 太模糊，更新什么？
❌ 说明文档.md          - 太通用
```

---

### 2. 时间性记录文档

**格式**: `YYYY-MM-DD_HHMM_[描述].md`

**适用于**:
- 阶段完成报告
- 重要事件记录
- 问题解决记录

**示例**:
```
✅ 2025-11-20_1615_后端开发完成报告.md
✅ 2025-11-20_1827_前端开发完成报告.md
✅ 2025-11-20_1400_Phase2进度报告_数据模型实现.md
```

---

### 3. 持续性文档

**格式**: `YYYY-MM-DD_[主题].md`

**适用于**:
- 持续更新的对话记录
- 每日工作日志

**示例**:
```
✅ 2025-11-20_完整对话记录.md
✅ 2025-11-20_开发日志.md
```

## ⚠️ 创建新文档前的检查清单

### Step 1: 检查是否已存在

```bash
# 列出所有Markdown文档
ls 项目记录/*.md
ls 项目记录/*/*.md

# 搜索关键词
grep -r "关键词" 项目记录/
```

### Step 2: 确认命名清晰

问自己三个问题：
1. ❓ 文件名是否明确表达内容？
2. ❓ 是否会与现有文档混淆？
3. ❓ 其他人看到名称能理解内容吗？

### Step 3: 选择合适位置

```
项目记录/
├── [规范性文档].md        ← 放在根目录
├── 过程记录/
│   └── [时间性记录].md    ← 放在过程记录
└── 聊天记录/
    └── [对话记录].md      ← 放在聊天记录
```

### Step 4: 更新索引

在 `项目记录/README.md` 中添加新文档的链接

## 🔍 文档分类指南

### 规范类 (Specifications)

**特征**: 定义规则、标准、流程

**命名**: `[主题]规范.md` 或 `[主题]管理规范.md`

**示例**:
- ✅ 聊天记录管理规范.md
- ✅ 代码提交规范.md
- ✅ 文档命名规范.md

---

### 说明类 (Instructions)

**特征**: 解释如何使用工具、脚本、功能

**命名**: `[主题]说明.md` 或 `[主题]使用说明.md`

**示例**:
- ✅ 状态自动更新说明.md
- ✅ 脚本使用说明.md
- ✅ API调用说明.md

---

### 指南类 (Guides)

**特征**: 详细的配置、部署、开发指导

**命名**: `[主题]指南.md`

**示例**:
- ✅ 局域网访问指南.md
- ✅ 部署指南.md
- ✅ 开发环境配置指南.md

---

### 记录类 (Records)

**特征**: 事件、过程、对话的记录

**命名**: `YYYY-MM-DD_[描述].md`

**示例**:
- ✅ 2025-11-20_完整对话记录.md
- ✅ 2025-11-20_1615_后端开发完成报告.md

## 🛠️ 文档维护

### 定期检查 (每月)

1. **查找重复**
   ```bash
   # 列出所有文档并检查名称相似的
   ls -lh 项目记录/*.md
   ```

2. **清理过时文档**
   - 标记为已废弃
   - 移动到 `archive/` 目录
   - 或直接删除（Git保留历史）

3. **更新索引**
   - 确保 `README.md` 索引是最新的
   - 移除已删除文档的链接

---

### 重命名文档

**步骤**:
```bash
# 1. 使用git mv保留历史
git mv "旧名称.md" "新名称.md"

# 2. 搜索所有引用并更新
grep -r "旧名称" .

# 3. 更新README索引

# 4. 提交更改
git commit -m "重命名文档：旧名称 → 新名称"
```

---

### 合并重复文档

**步骤**:
1. 比较内容，确定哪个是主文档
2. 将有用内容合并到主文档
3. 删除重复文档
4. 更新所有引用链接
5. 提交更改并说明原因

- [ ] 没有与现有文档重复
- [ ] 放在正确的目录下
- [ ] 已更新 `README.md` 索引
- [ ] 内容完整，格式统一
- [ ] 链接都能正常访问

## 🎯 最佳实践总结

### ✅ 应该做的

1. **明确命名** - 文件名体现主要内容
2. **分类清晰** - 规范/说明/指南/记录
3. **及时索引** - 更新README
4. **避免重复** - 创建前先检查
5. **保持简洁** - 一个文档一个主题

### ❌ 不应该做的

1. **模糊命名** - "文档.md", "说明.md"
2. **随意存放** - 不按分类存放
3. **忽略索引** - 不更新README
4. **重复创建** - 不检查已有文档
5. **内容混杂** - 一个文档多个主题
