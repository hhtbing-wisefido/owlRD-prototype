# 规则文件命名规范

**规则类型**: 🔴 强制规则  
**强制级别**: 🔴 严格执行  
**适用范围**: .windsurf/rules/ 目录下的所有规则文件  

---

## 📋 命名规范

### 标准格式

`
NN-中文名称.md
`

**格式说明**：
- NN - 两位数字编号（00-99）
- - - 连字符分隔
- 中文名称 - 清晰的中文描述
- .md - Markdown扩展名

---

### 核心规则（00-08）
`
00-核心工作原则.md
01-文件操作规范.md
02-目录管理规范.md
03-文件命名规范.md
04-Git工作流规范.md

07-严格目录控制.md

`

### 扩展规则（09-99）
`
09-项目配置.md
10-架构使用指南.md
11-规则同步说明.md
12-迁移检查清单.md
13-项目配置示例.md
14-规则文件命名规范.md
`

### 特殊文件
`
README.md   唯一不需要数字前缀的文件
`

## ❌ 错误示例

### 禁止的命名方式

`
❌ core-principles.md          # 缺少数字前缀
❌ 00-core-principles.md       # 使用英文而非中文
❌ 1-核心原则.md                # 数字不足两位
❌ 00_核心原则.md               # 使用下划线而非连字符
❌ 00 核心原则.md               # 使用空格而非连字符
❌ 核心原则.md                  # 缺少数字和连字符
❌ test.md                     # 非中文名且无编号
`

## 🎯 编号分配规则

### 编号范围

| 范围 | 用途 | 激活模式 |
|------|------|----------|
| **00-08** | 核心强制规则 | Always On |
| **09** | 项目特定配置 | Model Decision |
| **10-19** | 使用指南和文档 | Manual |
| **20-89** | 预留扩展 | 待定 |
| **90-98** | 临时测试规则 | Manual |
| **99** | 保留（特殊用途） | - |

### 编号原则

1. **核心规则优先** - 编号00-08保留给最重要的规则
2. **按重要性排序** - 数字越小优先级越高
3. **分类连续** - 同类规则使用连续编号
4. **预留空间** - 为未来扩展预留编号

## 📝 命名要求

### 中文名称规范

#### ✅ 应该使用
- **清晰具体** - 准确描述规则内容
- **简洁明了** - 通常2-6个汉字
- **专业术语** - 使用标准的技术术语

#### ❌ 不应使用
- 模糊名称（如"规则1"、"新规则"）
- 过长名称（超过10个汉字）
- 口语化表达（如"怎么做"、"注意事项"）

#### 示例对比

| ❌ 不好的命名 | ✅ 好的命名 | 说明 |
|--------------|------------|------|
| 规则1 | 核心工作原则 | 具体描述内容 |
| 一些文件的操作方法 | 文件操作规范 | 简洁专业 |
| 关于目录的说明 | 目录管理规范 | 明确范围 |
| 新建的规则 | 规则文件命名规范 | 清晰具体 |

## 🔄 重命名流程

### 如果需要重命名规则文件

1. **更新文件名**
   `ash
   Rename-Item "旧文件名.md" "新文件名.md"
   `

2. **更新frontmatter**
   `yaml
   ---
   title: "新的英文标题"
   description: "新的中文描述"
   trigger: always  # 或 manual/model_decision
   ---
   `

3. **更新引用**
   - 检查其他规则文件中的引用
   - 更新README.md中的索引
   - 更新config.json（如果存在）

4. **提交更改**
   `ash
   git add .windsurf/rules/
   git commit -m "refactor: 重命名规则文件 旧名  新名"
   `

- ✅ JSON格式，非Markdown
- ✅ 定义规则加载顺序

## 💡 最佳实践

### 1. 规划编号
创建新规则前，先规划好编号：
- 确定规则类别（核心/扩展/文档）
- 选择合适的编号范围
- 避免与现有规则冲突

### 2. 一致性
保持命名风格一致：
- 统一使用"规范"、"规则"、"指南"等术语
- 统一描述格式（如都用"XX规范"或"XX指南"）

### 3. 文档同步
创建规则后及时更新：
- README.md中添加索引
- 更新规则总数
- 记录版本变更
